## Command

Hierarchical root (traverses all 6 demo sub-sheets in one pass):

```
tools/bin/kicad-cli sch erc \
  --output board/kicad/e1-phone/erc/erc-report.json \
  --severity-all --format json \
  board/kicad/e1-phone/schematic/e1-phone-demo.kicad_sch

tools/bin/kicad-cli sch erc \
  --output board/kicad/e1-phone/erc/erc-report.txt \
  --severity-all --format report \
  board/kicad/e1-phone/schematic/e1-phone-demo.kicad_sch
```

The demo project is a single linked hierarchy: `e1-phone-demo.kicad_sch`
references the 6 block sheets (`power_usb-demo`, `compute-demo`,
`display_camera-demo`, `radios-demo`, `audio_buttons-demo`,
`split_interconnect-demo`) via `(sheet ...)` blocks, so one ERC run covers
all of them. Per-sheet runs were also executed for cross-check and each
returned `Found 0 violations`.

## Triage

The demo schematics (`schematic/*-demo.kicad_sch`, generated by
`scripts/generate_e1_phone_routed_mainboard_demo.py`) are deliberately
text-only annotation scaffolds. Each sheet carries a title block, an
`Evidence class: non_release_routing_demonstration` banner, and `(text "...")`
lines that describe the intended connectors, nets, and decoupling — but no
`(symbol ...)` instances, no wires, and no power ports. The routing
demonstration with real placeholder land patterns and named nets lives on the
PCB side (`pcb/e1-phone-mainboard-demo.kicad_pcb`), not in the schematic.

| Class | Item | Observed | Verdict |
|-------|------|----------|---------|
| (b) expected for non-release demo | Unconnected-pin warnings | None — no symbols/pins exist | No fix possible or needed; expected for a text-only demo |
| (b) expected for non-release demo | Missing power-flag (`PWR_FLAG`) warnings | None — no power nets are driven by symbols | Adding `PWR_FLAG` would be meaningless with no electrical net; not added |
| (b) expected for non-release demo | "No-ERC" / scaffold conductors | None — no wires/labels resolve to nets | Expected; no no-connect flags applicable |
| (a) real issue to note | ERC silence is a property of an empty-electrical schematic, **not** evidence of a verified design | n/a | Documented: a 0-violation result here proves the demo hierarchy parses and links cleanly, **not** that the e1-phone design is electrically correct |

**No trivial fixes were applied.** The standard non-release demo trivia
(label no-connects, add `PWR_FLAG`) presuppose symbol/net content that this
text-only scaffold does not have. Adding placeholder symbols purely to host a
`PWR_FLAG` would inflate the schematic symbol/text counts tracked in
`schematic-symbol-footprint-closure.yaml` without adding real evidence, so it
was correctly avoided.

## Verdict

**Demo ERC clean modulo documented non-release waivers.** The demo schematic
hierarchy links and ERC-parses with zero errors and zero warnings under KiCad
9.0.9. This is `non_release_demo_erc` evidence only: it confirms structural
integrity of the demo hierarchy, not electrical correctness of the e1-phone
design.

### Not closed / still blocked

This run does **not** satisfy production ERC and does **not** flip any release
or closure gate. `schematic-symbol-footprint-closure.yaml` keeps
`erc_report_present: false`, `status:
blocked_requires_real_kicad_symbols_supplier_footprints_and_erc`, and the
`erc_clean` forbidden-claim — all of which remain in force.

### Evidence required for production ERC

Production ERC becomes meaningful only once the text-only scaffold is replaced
with reviewed symbol capture:

1. Real KiCad symbols with correct electrical pin types (power-in/out,
   bidirectional, passive, no-connect) for every refdes — most importantly the
   **E1 SoC** (`U_SOC`) full BGA pinout, `U_LPDDR_UFS`, `U_PMIC`, the USB-C
   receptacle, and all FPC/B2B connectors with supplier-locked pin order.
2. Real wires/labels/buses tying the nets in `block-netlist.yaml`, plus
   `PWR_FLAG` on externally-driven power nets and explicit no-connect flags on
   intentionally floating pins.
3. Footprint and 3D-model bindings (out of ERC scope but gated alongside).

Then run, on the **production** root schematic (not the `-demo` set):

```
tools/bin/kicad-cli sch erc \
  --output board/kicad/e1-phone/erc/erc-report-release.json \
  --severity-all --format json \
  board/kicad/e1-phone/schematic/e1-phone.kicad_sch
```

A production ERC pass = 0 errors with every warning either eliminated or
explicitly waived with written justification, against symbols that carry the
full SoC pinout. Only that result may set `erc_report_present: true` and
release the `erc_clean` gate.
