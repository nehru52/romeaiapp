=== WHY IT IS BLOCKED (real blocker, not slowness) ===
At guest uptime ~38-40s TWO vendor HAL services SIGABRT (signal 6) on service
registration and never recover:

  android.hardware.boot-service.default:
    "Check failed: status == STATUS_OK (status=-3, STATUS_OK=0)
     Failed to add service android.hardware.boot.IBootControl/default -3"

  vendor.keymint-rust (android.hardware.security.keymint-service.rust):
    "HAL service failed: Failed to register service
     android.hardware.security.keymint.IKeyMintDevice/default: UNKNOWN_ERROR"

With KeyMint HAL dead, keystore2 cannot publish android.security.maintenance, so
vold spins forever trying to start it (servicemanager retry storm: 11,946 occurrences;
init: "Too many pending control messages, dropped 'interface_start' for
'aidl/android.security.maintenance'"). system_server never finishes, sys.boot_completed
never reaches 1, adbd never starts.

Observed end state at uptime ~1656s (~27.6 min wall): identical vold/keystore retry
loop, adb host port 6520 NOT listening, `adb connect` => Connection refused, device
absent from `adb devices`. This is terminal, not slow TCG progress.

Root cause: host-package / guest version skew at the HAL VINTF boundary (caveat #1).
The Eliza partitions are android16-qpr2 (BP4A.251205.006); the prebuilt cvd host package
+ secure_env are stock android16 build 15357239. servicemanager rejects the eliza vendor
HAL interface registrations (boot status=-3 BAD_TYPE; keymint UNKNOWN_ERROR). The same
host package boots the STOCK aosp_cf_riscv64_phone images deep into init/system_server
(prior runs), so the host path itself works — the failure is specific to the eliza
vendor HAL surface interacting with this host package's framework/secure_env.
