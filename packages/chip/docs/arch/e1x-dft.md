# E1X Memory-Integrity and DFT Strategy

This document is the design-for-test (DFT) and memory-integrity strategy for
the E1X wafer-scale mesh (a logical 32x32 array of processing elements with 2
spare rows and 2 spare columns, each PE owning 48 KiB of local SRAM). It
defines the scan-chain insertion plan, the MBIST distribution model, the ECC
policy, and how memory test interacts with the existing wafer-sort and
repair-ROM flow. It states explicitly which artifacts are RTL-proven in this
package and which require the foundry scan/ATPG and silicon flows that are not
runnable here (the fail-closed boundary).

The companion RTL is `rtl/e1x/e1x_sram_ecc.sv` and `rtl/e1x/e1x_mbist.sv`,
verified by `verify/cocotb/e1x_dft/` and gated by
`scripts/check_e1x_dft_cocotb.py`.

## Scope and Fail-Closed Boundary

RTL-proven in this package (executable evidence in cocotb):

- SECDED encode/decode datapath: every single-bit codeword flip is corrected,
  every double-bit flip is detected without miscorrection, and the corrected/
  detected status counters advance and clear.
- March C- MBIST sequencer: it walks the address space, passes a clean SRAM
  model, and detects an injected stuck-at fault (both polarities) with the
  correct failing address and failing bit index.

NOT proven here, and fail-closed gated as external dependencies:

- Scan-chain stitching, scan-cell insertion, and ATPG pattern generation.
  These require the foundry standard-cell library, a scan-aware synthesis flow
  (DFT compiler), and an ATPG tool with the PDK fault models. None of those
  inputs exist in this package, so no scan coverage number is claimed.
- SRAM macro behavior (sense-amp timing, redundancy columns, BIST collar
  generated by the memory compiler). The behavioral SRAM in `e1x_mbist.sv` is
  a placeholder for the foundry macro and is replaced in the PD flow.
- At-speed test, IDDQ, and burn-in. These are silicon/test-floor activities.
- Full-wafer fault coverage and defect density correlation. This requires
  wafer-sort data from fabricated silicon.

Any claim about scan coverage percentage, ATPG pattern count, at-speed
transition coverage, or measured silicon defect rate is BLOCKED until the
foundry DFT/ATPG flow and silicon are available, and must be recorded as a
gate or evidence artifact rather than asserted in prose.

## ECC Policy

Which memories are covered:

- PE local SRAM (48 KiB/PE) is the primary ECC target. It holds quantized model
  weights and activations streamed in via the shard loader
  (`rtl/e1x/e1x_local_sram_shard_loader.sv`); a silent bit flip there corrupts
  inference output, so it is protected with SECDED.
- The repair ROM and route-table SRAM are protected by parity at minimum;
  because a repair-descriptor flip remaps a tile incorrectly, the repair-ROM
  loader (`rtl/e1x/e1x_repair_rom_loader.sv`) already validates a magic header
  and bounded counts, and parity is added at integration as defense in depth.

Code and correction scope:

- The code is an extended Hamming code over a 32-bit data word: 6 Hamming
  parity bits (single-error correct) plus 1 overall parity bit (double-error
  detect), giving 7 check bits and a 39-bit stored codeword. This is SECDED:
  single-error-correct, double-error-detect.
- Correction scope is per word. A single-bit error per 32-bit word is corrected
  inline on read; a double-bit error in the same word is detected and reported
  but not corrected (it raises a detected-double-error event). Errors spanning
  more than two bits in one word are outside the code's guarantee and may
  alias; word-level interleaving in the macro layout keeps physically adjacent
  cells in different logical words so a multi-cell particle strike maps to
  single-bit-per-word errors.
- Status counters (corrected-error count, detected-double-error count) are
  exposed per ECC instance and saturate at 32 bits. They are read by firmware
  to track in-field bit-error rate and to escalate a tile to the repair flow if
  the corrected-error rate crosses a threshold.

The ECC width can be reparameterized to 64-bit data (Hamming(72,64) +1 overall
parity = 8 check bits) for arrays where the access width is 64-bit; the codec
in `e1x_sram_ecc.sv` is parameterized on `DATA_BITS` and `PARITY_BITS`.

## MBIST Algorithm

The MBIST controller implements March C- (sometimes written March C minus),
the standard ten-operation march test reduced to six elements:

1. M0: ascending, write 0 to every cell (background).
2. M1: ascending, read 0 then write 1.
3. M2: ascending, read 1 then write 0.
4. M3: descending, read 0 then write 1.
5. M4: descending, read 1 then write 0.
6. M5: descending, read 0.

This covers stuck-at faults (SAF), transition faults (TF), and a class of
coupling faults (CFin, CFid) by virtue of the alternating ascending/descending
read-then-write order. At signoff the controller is run with multiple data
backgrounds (solid 0/1 and checkerboard/inverse) to extend coupling-fault and
neighborhood-pattern-sensitive-fault coverage; the RTL-proven core here is the
address sequencer and pass/fail/failing-address reporting over one background.

On a mismatch the controller latches the failing word address, the first
failing bit index, the expected value, and the actual value, then asserts
`fail` and `done`. Firmware reads the failing address to decide whether the
defect is repairable by spare remapping (whole-tile granularity) or whether the
tile is marked dead.

## MBIST Distribution Across the Mesh

Per-tile vs shared controller:

- The mesh has up to 34x34 = 1156 physical tiles, each with one local SRAM
  macro. A single shared MBIST controller sequenced across all tiles would
  serialize 1156 march runs and make wafer-sort test time the dominant cost; a
  full per-tile controller replicated 1156 times wastes area on logic that runs
  only at test time.
- The chosen model is a hierarchical, banded distribution: one MBIST controller
  per tile row (34 controllers), each driving the local SRAM macros of the
  tiles in its row over a shared test bus, tiles tested in parallel across rows.
  This bounds test time at one row-length march and keeps the per-controller
  area amortized across a row.
- The controllers share a single test-access port (TAP)-driven start/done
  handshake and an OR-reduction of per-row `fail` plus a scan-out of the
  per-row failing-address registers, so the tester collects the full
  failing-tile map in one pass.

The single-instance controller in `e1x_mbist.sv` is the unit that is replicated
per row; this package proves the unit, not the 34-way replication or the TAP
integration (those depend on the foundry TAP/JTAG macro).

## Scan-Chain Stitching Plan

Tile-level scan:

- Each PE's sequential elements (core pipeline registers, router queue
  pointers, repair-state registers) are converted to scan flops and stitched
  into one tile-local scan chain during scan-aware synthesis. The tile exposes
  scan-in, scan-out, and scan-enable on its boundary.
- Tile scan chains are concatenated along a row into a row chain, and row chains
  are selected by the TAP into a small number of top-level chains (balanced for
  equal length so shift time is uniform). The spare rows/columns are included in
  the chains so a repaired mesh remains fully scannable.
- Memory macros are bypassed in the logic scan chains and tested by MBIST
  instead; the MBIST collar's own registers are on a dedicated short chain so
  the tester can configure background/algorithm without shifting the full logic
  chain.

Boundary and at-speed:

- Boundary scan (IEEE 1149.1) on the I/O ring supports board-level
  interconnect test and is driven by the same TAP.
- At-speed (transition-fault) ATPG uses launch-on-capture with the functional
  clock; the clock-control logic for this is generated by the DFT flow, not by
  this RTL.

Scan stitching, chain balancing, and ATPG pattern generation are foundry-flow
activities. No scan-chain netlist or coverage figure is produced in this
package; that is the fail-closed boundary stated above.

## Repair Interaction with Wafer-Sort and the Repair-ROM Flow

- Wafer sort runs MBIST per tile and collects the failing-tile map from the
  per-row failing-address scan-out. A tile with an unrepairable SRAM fault (or a
  logic fault found by ATPG) is recorded as defective in the defect map.
- The defect map is consumed by the existing repair flow: spare rows/columns
  remap defective logical tiles to physical spares, and the resulting remap and
  route descriptors are written into the repair ROM. The repair-ROM loader
  (`rtl/e1x/e1x_repair_rom_loader.sv`) parses these at boot and programs the
  repair-aware routers (`rtl/e1x/e1x_repair_aware_router.sv`,
  `e1x_repair_route_table.sv`).
- ECC and MBIST are complementary: MBIST plus spare remapping handles hard
  (manufacturing) defects at sort; ECC handles soft (in-field, transient or
  marginal) bit errors during operation. The ECC corrected-error counter feeds
  an in-field escalation path: a tile whose corrected-error rate is sustained
  above threshold can be remapped to a spare on the next boot by updating the
  repair ROM, reusing the same repair descriptors that wafer sort produces.
