# @elizaos/security — SOC2 Control Surface

`@elizaos/security` is the canonical KMS interface for the Eliza runtime. It is the single boundary through which Confidential and Restricted-class data is encrypted, decrypted, signed, and verified.

This document describes the SOC2 controls this package implements or supports.

## Trust Service Criteria served

| TSC | How this package contributes |
|---|---|
| **C1.1** Encryption at rest | Provides AES-256-GCM envelope encryption with mandatory AAD for all Confidential / Restricted data persisted by Cloud and runtime. |
| **C1.2** Secure disposal | DEK-destroy operations are exposed as a non-reversible API; emits an `audit_events` row. |
| **CC6.7** Encryption in transit | Provides verified HMAC-SHA256 helpers and Ed25519 signing primitives used by webhook ingress and plugin manifest verification. |
| **CC6.8** Integrity | DSPy prompt HMAC verification (`OptimizedPromptService`) and plugin manifest verification ride this package's primitives. |

## Design invariants

1. **AAD is mandatory.** The encrypt API rejects callers that fail to supply AAD. AAD is bound to context — typically `namespace|user_id|field`. Callers that need ciphertext stability across renames must thread the right AAD; the package does not provide a "no-AAD" mode.
2. **Keys never leave the boundary in plaintext.** The package interfaces with the configured KMS backend (Steward in production; Vault acceptable for self-host); wrapped DEKs cross the boundary, plaintext keys do not.
3. **Algorithms are policy.** Only the algorithms named in [`../../../../POLICIES/12-cryptography.md`](../../../../POLICIES/12-cryptography.md) are exposed. Adding a new algorithm requires a policy PR.
4. **Audit-on-use.** Every key-material operation (issue, wrap, unwrap, destroy, sign) is observable for the audit-event sink at the boundary. Callers do not need to remember to log.

## Rotation responsibilities

Rotation is driven by Steward (or the configured backend) per the schedule in [`../../../../docs/security/KEY-LIFECYCLE.md`](../../../../docs/security/KEY-LIFECYCLE.md). This package:

- Accepts both old and new key during the dual-accept window when unwrapping.
- Re-emits wrap operations under the current KEK on the next write.
- Exposes a `getActiveKid()` and `getDecryptKidSet()` API so callers can detect stale ciphertext.

## Disposal contract

- `destroyKey(kid, reason)` is non-reversible.
- Emits an `audit_events` row with operator, reason, and timestamp.
- Returns only after the underlying backend confirms destruction.

## Evidence an auditor will see

- Source-control evidence of the AAD-mandatory check (test coverage).
- Steward audit log sample showing wrap / unwrap / destroy with operator + AAD.
- `audit_events` rows generated by `destroyKey` calls during DSR / retention runs.
- CI dependency-audit + scanner output for this package.
